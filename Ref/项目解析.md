## 一、项目概述

### 1.1 什么是 ESP-Claw

ESP-Claw 是乐鑫（Espressif）开发的一款**面向 IoT 设备的 AI Agent 框架**，运行在 ESP32-S3 芯片上。它实现了一种"Chat Coding"范式——用户通过 IM 聊天对话来定义设备行为，设备使用大语言模型（LLM）动态生成并执行 Lua 脚本、控制外设，同时管理自身的结构化记忆。

### 1.2 核心特性

| 特性 | 说明 |
|------|------|
| **边缘 AI** | Agent Loop（感知→推理→决策→执行）完全在 ESP32 芯片本地运行，不依赖云端 |
| **Chat Coding** | 非程序员也能通过 IM 聊天（Telegram、微信、飞书、QQ）结合动态 Lua 加载来定义设备行为 |
| **事件驱动** | 任何事件（传感器数据、定时器、IM 消息）都可以触发 Agent Loop，而非仅限于用户消息 |
| **结构化记忆** | 设备端长期记忆包含 profile、preferences、facts、events、rules 五种类型，以 JSONL 格式存储，支持标签检索 |
| **MCP 双身份** | 设备既可作为 MCP Server（暴露硬件能力为工具），也可作为 MCP Client（调用其他 MCP 服务） |
| **模块化** | 每个能力模块都可以在构建时选择包含或排除 |

### 1.3 技术规格

- **目标芯片**: ESP32-S3（需要 PSRAM）
- **许可证**: Apache-2.0
- **构建框架**: ESP-IDF 5.5.1 + CMake
- **脚本引擎**: Lua 5.x（嵌入式脚本运行时）
- **推荐 LLM**: GPT-5.4、Qwen3.5-plus 等强推理模型
- **IM 平台**: Telegram、微信、飞书、QQ

## 三、四层架构总览

ESP-Claw 采用严格的**四层架构**设计，自上而下分别为：

```
┌──────────────────────────────────────────────────────┐
│  Layer 1: 应用装配层 (Application Assembly)           │
│  application/basic_demo/main/                         │
│  main.c → app_claw.c → 各子系统初始化与装配           │
├──────────────────────────────────────────────────────┤
│  Layer 2: 能力插件层 (Capability Plugins)             │
│  components/claw_capabilities/                        │
│  18 个独立的 cap_* 模块，每个注册自己的能力描述符      │
├──────────────────────────────────────────────────────┤
│  Layer 3: 运行时核心层 (Runtime Core)                 │
│  components/claw_modules/                             │
│  claw_core / claw_cap / claw_event_router /           │
│  claw_memory / claw_skill                             │
├──────────────────────────────────────────────────────┤
│  Layer 4: 设备/脚本扩展层 (Device/Script Extensions)  │
│  components/lua_modules/                              │
│  18 个 Lua 模块，暴露硬件外设到 Lua VM               │
└──────────────────────────────────────────────────────┘
```

### 各层职责

| 层级 | 目录 | 职责 |
|------|------|------|
| **Layer 1** | `application/basic_demo/main/` | ESP-IDF 入口点，初始化 NVS、FAT、WiFi，装配所有核心模块、能力插件和上下文提供者 |
| **Layer 2** | `components/claw_capabilities/` | 自包含的能力模块，每个注册 `claw_cap_group_t`，包含生命周期回调和 LLM 工具调用 JSON Schema |
| **Layer 3** | `components/claw_modules/` | 五大核心模块：Agent Loop、能力注册表、事件路由、结构化记忆、技能管理 |
| **Layer 4** | `components/lua_modules/` | 硬件外设的 Lua 绑定，将 GPIO/I2C/SPI/UART 等暴露为 Lua 函数 |

### 数据流总览

```
用户消息 (IM/串口/Web)
    │
    ▼
事件路由器 (claw_event_router) ─── 规则匹配 ───→ 执行动作
    │                                              │
    │  (路由到 Agent)                               │ (直接调用能力/运行脚本)
    ▼                                              │
Agent Loop (claw_core)                              │
    │                                               │
    ├── 收集上下文 (记忆/技能/工具列表)              │
    ├── 调用 LLM                                    │
    ├── LLM 返回工具调用 ──→ claw_cap 调度执行 ──────┘
    ├── 工具结果反馈回 LLM → 循环迭代（最多 20 次）
    └── LLM 返回文本 → 最终响应 → 事件路由 → IM 发送
```

## 四、核心模块详解

### 4.1 claw_core — Agent Loop 运行时

**位置**: `components/claw_modules/claw_core/`

这是整个框架的心脏，实现了"感知→推理→决策→执行"的 Agent Loop。

#### 核心源文件

| 文件 | 作用 |
|------|------|
| `src/claw_core.c` | Agent Loop 主循环：请求队列、LLM 迭代、工具调用、上下文组装 |
| `src/claw_core_llm.c` | LLM 门面：初始化、聊天、媒体处理的上层接口 |
| `src/claw_task.c` | 任务封装：支持 PSRAM 栈分配的任务创建 |
| `src/llm/claw_llm_runtime.c` | 后端分发：根据配置选择 LLM 后端（vtable） |
| `src/llm/claw_llm_http_transport.c` | HTTP 客户端：支持中断的 HTTP 通信 |
| `src/llm/media/claw_media_pipeline.c` | 媒体管道：图片准备（base64、URL） |
| `src/llm/backends/claw_llm_backend_openai_compatible.c` | OpenAI 兼容后端（OpenAI、Qwen 等） |
| `src/llm/backends/claw_llm_backend_anthropic.c` | Anthropic Claude 后端 |
| `src/llm/backends/claw_llm_backend_custom.c` | 可插拔的自定义后端 |

#### Agent Loop 工作流程

```
1. 从请求队列接收新请求 (claw_core_submit)
2. 收集所有注册的上下文提供者内容:
   ├── 记忆档案 (profile memory)
   ├── 长期记忆 (long-term memory)
   ├── 会话历史 (session history)
   ├── 技能列表 (skills list)
   ├── 激活技能的文档 (active skill docs)
   ├── LLM 可见工具列表 (tools JSON)
   ├── Lua 异步任务状态 (async jobs)
   └── 时间上下文 (time context)
3. 构建系统提示词 (system prompt + 所有上下文)
4. 调用 LLM API
5. 解析 LLM 响应:
   ├── 如果返回工具调用 (tool_calls):
   │   ├── 通过 claw_cap_call() 执行对应能力
   │   ├── 将工具结果追加到消息历史
   │   └── 回到步骤 4（最多迭代 20 次）
   └── 如果返回纯文本:
       └── 这是最终响应 → 发布 out_message 事件
6. 发布响应事件 → 事件路由器分发给 IM 通道
```

#### 关键 API

```c
// 初始化和启动
claw_core_err_t claw_core_init(const claw_core_config_t *config);
claw_core_err_t claw_core_start(void);

// 提交请求
claw_core_err_t claw_core_submit(claw_core_request_t *request);

// 上下文提供者注册
void claw_core_register_context_provider(const char *name, claw_context_provider_t provider);

// 中止当前请求
void claw_core_abort(void);
```

#### 任务栈策略

- `claw_core` 任务栈: 16KB（从 PSRAM 分配）
- 事件路由器任务栈: 8KB（从 PSRAM 分配）
- 使用 `CLAW_TASK_STACK_PREFER_PSRAM` 标志通过 `claw_task_create()` 在 PSRAM 可用时优先从 PSRAM 分配栈空间

#### LLM 后端支持

| 后端 | 说明 | 配置项 |
|------|------|--------|
| `openai_compatible` | OpenAI、Qwen (DashScope) 及所有兼容 API | API Key、Base URL、Model |
| `anthropic` | Anthropic Claude Messages API | API Key、Model |
| `custom` | 可通过 `claw_llm_register_custom_backend()` 注册 | 自定义 |

### 4.2 claw_cap — 能力注册与调度

**位置**: `components/claw_modules/claw_cap/`

能力注册表，管理所有能力组和能力描述符的动态表，控制 LLM 可见性。

#### 核心数据结构

```c
// 能力描述符
typedef struct {
    const char *id;              // 唯一标识
    const char *name;            // 显示名称
    const char *family;          // 所属族
    const char *description;     // 描述文本
    claw_cap_kind_t kind;        // CALLABLE / EVENT_SOURCE / HYBRID
    claw_cap_flags_t cap_flags;  // CALLABLE_BY_LLM / EMITS_EVENTS / ...
    const char *input_schema_json; // JSON Schema（LLM 工具调用用）
    // 生命周期回调
    claw_cap_init_cb_t init;
    claw_cap_start_cb_t start;
    claw_cap_stop_cb_t stop;
    claw_cap_execute_cb_t execute;
} claw_cap_descriptor_t;

// 能力组
typedef struct {
    const char *group_id;
    const char *description;
    claw_cap_descriptor_t *descriptors;
    int descriptor_count;
} claw_cap_group_t;
```

#### 能力类型 (kind)

| 类型 | 说明 |
|------|------|
| `CALLABLE` | 可被 LLM 调用的工具（如发消息、运行脚本） |
| `EVENT_SOURCE` | 事件源，主动产生事件（如 IM 消息接收） |
| `HYBRID` | 既是事件源也可被调用（如 Lua 运行时） |

#### 能力标志 (cap_flags)

| 标志 | 说明 |
|------|------|
| `CALLABLE_BY_LLM` | LLM 可调用此能力 |
| `EMITS_EVENTS` | 此能力会产生事件 |
| `SUPPORTS_LIFECYCLE` | 支持启动/停止生命周期 |
| `RESTRICTED` | 受限能力，需要特殊权限 |

#### 生命周期状态机

```
REGISTERED → STARTED → DISABLED → DRAINING → UNLOADING
                ↑________↓
```

- **REGISTERED**: 已注册但未启动
- **STARTED**: 正常运行中
- **DISABLED**: 已禁用，不再接受新调用
- **DRAINING**: 正在排空进行中的调用
- **UNLOADING**: 即将卸载

#### 会话感知可见性

LLM 工具可见性按会话控制。当技能被激活时，其关联的能力组仅在该会话内可见，实现多用户隔离。

### 4.3 claw_event_router — 事件路由引擎

**位置**: `components/claw_modules/claw_event_router/`

事件驱动的路由引擎，运行在独立的 FreeRTOS 任务上，通过队列接收事件。

#### 核心源文件

| 文件 | 作用 |
|------|------|
| `src/claw_event.c` | 事件结构体的创建、克隆、释放 |
| `src/claw_event_router.c` | 规则引擎：匹配、模板渲染、6 种动作类型 |

#### 事件结构

```c
typedef struct {
    char *event_type;        // 事件类型（message, trigger 等）
    char *source_cap;        // 来源能力
    char *source_channel;    // 来源通道（tg, wechat, feishu, qq）
    char *chat_id;           // 聊天 ID
    char *content_type;      // 内容类型（text, image, file）
    char *text;              // 文本内容
    char *session_id;        // 会话 ID
    // ... 更多字段
} claw_event_t;
```

#### 路由规则

规则从 JSON 文件加载，支持以下匹配条件：

| 匹配字段 | 说明 |
|----------|------|
| `event_type` | 事件类型（message, trigger） |
| `source_cap` | 来源能力 ID |
| `source_channel` | 来源通道 |
| `chat_id` | 聊天 ID |
| `content_type` | 内容类型 |
| `text` | 文本内容（支持正则） |

#### 6 种动作类型

| 动作 | 说明 |
|------|------|
| `call_cap` | 直接调用一个能力 |
| `run_agent` | 提交给 Agent Loop 处理 |
| `run_script` | 运行 Lua 脚本 |
| `send_message` | 发送消息到指定通道 |
| `emit_event` | 发出新事件（事件链） |
| `drop` | 丢弃事件 |

#### 模板渲染

动作参数支持模板变量渲染，使用 `{{event.xxx}}` 语法：

```json
{
    "action": "send_message",
    "params": {
        "channel": "{{event.source_channel}}",
        "chat_id": "{{event.chat_id}}",
        "text": "{{agent_response}}"
    }
}
```

#### 默认路由规则

`fatfs_image/router_rules/router_rules.json` 定义了默认规则：

1. 工作回复 → Agent → 交付响应
2. 如果没有规则匹配 "message" 事件且 `default_route_messages_to_agent` 为 true，自动路由到 Agent

### 4.4 claw_memory — 结构化记忆系统

**位置**: `components/claw_modules/claw_memory/`

为 Agent 提供持久化的结构化记忆，支持五种记忆类型。

#### 核心源文件

| 文件 | 作用 |
|------|------|
| `src/claw_memory.c` | 顶层初始化和回调注册 |
| `src/claw_memory_cap.c` | 记忆能力的注册（store/recall/update/forget） |
| `src/claw_memory_extract.c` | 异步 LLM 驱动的记忆提取 |
| `src/claw_memory_lightweight.c` | 轻量级记忆模式 |
| `src/claw_memory_profile.c` | 档案记忆（identity、soul、user） |
| `src/claw_memory_session.c` | 会话历史管理 |
| `src/claw_memory_storage.c` | JSONL/索引/摘要的持久化 |
| `src/claw_memory_utils.c` | 工具函数 |

#### 五种记忆类型

| 类型 | 说明 | 存储位置 |
|------|------|----------|
| **profile** | Agent 的身份、人格、用户档案 | `soul.md`, `identity.md`, `user.md` |
| **preference** | 用户偏好 | `memory_records.jsonl` |
| **fact** | 事实性知识 | `memory_records.jsonl` |
| **event** | 重要事件记录 | `memory_records.jsonl` |
| **rule** | 行为规则 | `memory_records.jsonl` |

#### 记忆存储格式

**JSONL 格式** (`memory_records.jsonl`):

```jsonl
{"id": "mem_001", "type": "fact", "tags": ["weather", "location"], "content": "用户居住在北京", "timestamp": "2025-01-15T10:30:00Z"}
{"id": "mem_002", "type": "preference", "tags": ["language"], "content": "用户偏好中文回复", "timestamp": "2025-01-15T10:35:00Z"}
```

**标签索引** (`memory_index.json`):

```json
{
    "weather": ["mem_001"],
    "location": ["mem_001"],
    "language": ["mem_002"]
}
```

#### 两种记忆模式

| 模式 | 说明 |
|------|------|
| **完整模式 (Full)** | 5 种结构化记忆类型 + 异步 LLM 提取 + 完整上下文提供 |
| **轻量模式 (Lightweight)** | 仅 profile + 轻量级长期记忆，适合资源受限场景 |

#### 记忆提取流程

```
每次对话回合结束后:
1. 将对话内容发送给 LLM
2. LLM 分析是否需要提取/更新/删除记忆
3. LLM 返回记忆操作指令
4. 系统执行记忆更新（写入 JSONL、更新索引）
```

#### 上下文提供

记忆系统注册了多个上下文提供者，供 Agent Loop 在每次 LLM 调用前收集：

- **profile_memory**: 提供 soul/identity/user 档案
- **long_term_memory**: 提供长期记忆（完整或轻量）
- **session_history**: 提供会话历史记录

### 4.5 claw_skill — 技能元数据与激活

**位置**: `components/claw_modules/claw_skill/`

技能是 Markdown 文档，描述如何使用某个特定能力。技能可以在运行时被动态激活/停用。

#### 核心源文件

| 文件 | 作用 |
|------|------|
| `src/claw_skill.c` | 技能加载、会话状态管理、停用守卫 |

#### 技能结构

```
技能 = Markdown 文档 (.md)
├── 标题和描述
├── 使用说明
├── 可用 API 列表
├── 示例用法
└── 关联的能力组
```

#### 技能生命周期

```
加载 (从 FAT 文件系统) → 列表展示 → 按会话激活/停用
```

- 技能文件存储在 FAT 文件系统的 `skills/` 目录
- 技能目录在 `skills_list.json` 中定义
- 激活技能会动态添加 LLM 可见的能力组
- 停用守卫防止不安全的停用（如 Lua 异步任务运行中）

#### 技能目录格式 (`skills_list.json`)

```json
{
    "skills": [
        {
            "name": "weather",
            "file": "weather.md",
            "caps": ["cap_web_search"]
        }
    ]
}
```

#### 会话隔离

技能激活状态按会话 (session) 隔离。当某个会话激活一个技能时：
1. 技能关联的能力组变为该会话可见
2. 其他会话不受影响
3. 会话结束时自动停用

## 八、设计模式与约定

### 8.1 能力注册模式 (Capability Registration)

每个能力模块暴露一个 `cap_xxx_register_group()` 函数，内部调用 `claw_cap_register_group()` 并传入静态定义的描述符数组。

```c
// 示例：注册能力组
void cap_lua_register_group(void) {
    static claw_cap_descriptor_t descs[] = {
        {
            .id = "lua_run",
            .name = "Run Lua Script",
            .kind = CAP_KIND_CALLABLE,
            .cap_flags = CAP_FLAG_CALLABLE_BY_LLM,
            .input_schema_json = "{\"type\":\"object\",\"properties\":{\"code\":{\"type\":\"string\"}}}",
            .execute = cap_lua_execute_run,
        },
        // ... 更多描述符
    };
    claw_cap_register_group(&(claw_cap_group_t){
        .group_id = "cap_lua",
        .descriptors = descs,
        .descriptor_count = sizeof(descs) / sizeof(descs[0]),
    });
}
```

### 8.2 上下文提供者模式 (Context Provider)

`claw_core` 使用可插拔的上下文提供者系统。每个提供者有名称和 `collect()` 函数，返回系统提示文本、消息历史或工具 JSON。

```c
// 注册上下文提供者
claw_core_register_context_provider("memory_profile", memory_profile_provider);
claw_core_register_context_provider("session_history", session_history_provider);
claw_core_register_context_provider("tools", tools_provider);
```

### 8.3 事件驱动架构 (Event-Driven)

事件路由器解耦了事件生产者（IM 能力、调度器、MCP、Lua）和消费者（Agent、IM 发送、Lua 脚本）。规则从 JSON 加载，支持模板变量渲染。

```
生产者 → 事件队列 → 规则匹配 → 动作执行 → 消费者
```

### 8.4 Agent Loop 模式

核心任务运行迭代循环，最多 20 次工具调用迭代：

```
收集上下文 → 调用 LLM → 工具调用? → 执行工具 → 回到 LLM
                              └─ 纯文本? → 最终响应 → 发布事件
```

### 8.5 记忆标签检索模式

使用摘要标签机制代替向量嵌入。每条记忆有 1-3 个关键词标签，LLM 选择性召回完整内容。这避免了在 ESP32 上运行向量数据库。

```
记忆条目 → 提取标签 → 建立索引 → LLM 选择标签 → 召回完整内容
```

### 8.6 命名约定

| 类别 | 前缀 | 示例 |
|------|------|------|
| 核心模块 | `claw_*` | `claw_core`, `claw_cap`, `claw_memory` |
| 能力插件 | `cap_*` | `cap_lua`, `cap_im_tg`, `cap_mcp_client` |
| Lua 模块 | `lua_module_*` | `lua_module_display`, `lua_module_camera` |
| 应用文件 | `basic_demo_*` / `app_*` | `basic_demo_settings`, `app_claw` |

### 8.7 文件组织约定

每个组件遵循 ESP-IDF 标准结构：

```
component_name/

```

## 九、总结

### 架构亮点

1. **全栈嵌入式 AI Agent**: 从底层硬件驱动到上层 LLM 交互，完整实现了 Agent Loop 框架
2. **高度模块化**: 5 个核心模块 + 18 个能力插件 + 18 个 Lua 模块，可按需裁剪
3. **事件驱动**: 通过事件路由器实现松耦合，任何事件都能触发 Agent 或执行动作
4. **Chat Coding 范式**: 用户无需编程，通过聊天即可定义设备行为
5. **结构化记忆**: 5 种记忆类型 + 标签检索，在 MCU 有限的资源下实现了有效的长期记忆
6. **MCP 双身份**: 既提供服务也消费服务，实现设备间协作

### 技术栈总结

```
硬件层:    ESP32-S3 + PSRAM + Flash (8/16MB)
系统层:    FreeRTOS + ESP-IDF 5.5.1 + FAT + NVS + OTA
核心层:    claw_core + claw_cap + claw_event_router + claw_memory + claw_skill
能力层:    18 个 cap_* 插件（IM/Lua/MCP/Scheduler/...）
脚本层:    18 个 lua_module_* 硬件绑定 + Lua 5.x VM
通信层:    HTTP + WebSocket + mDNS + WiFi STA/AP
AI 层:     OpenAI Compatible / Anthropic / Custom LLM 后端
用户层:    Web 配置 + 串口 CLI + 4 种 IM 平台
```
