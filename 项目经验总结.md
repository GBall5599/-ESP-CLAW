# ESP-CLAW 项目经验总结

本文档汇总 ESP-CLAW 移植过程中积累的技术经验和踩坑教训，供后续开发参考。

---

## 一、ESP-IDF v5.5.1 I2C 驱动迁移

### 背景

ESP-IDF v5.x 引入了全新的 I2C 主机驱动 API (`driver/i2c_master.h`)，与旧版 API (`driver/i2c.h`) 不兼容。v5.5.x 中**两者不能在同一 I2C 端口混用**，否则运行时直接 abort。

### 新旧 API 对照

| 操作 | 旧 API (`driver/i2c.h`) | 新 API (`driver/i2c_master.h`) |
|------|------------------------|-------------------------------|
| 初始化总线 | `i2c_param_config()` + `i2c_driver_install()` | `i2c_new_master_bus()` |
| 添加设备 | 无（用地址区分） | `i2c_master_bus_add_device()` |
| 写数据 | `i2c_master_write_to_device(bus, addr, ...)` | `i2c_master_transmit(dev_handle, ...)` |
| 读数据 | `i2c_master_write_read_device(bus, addr, ...)` | `i2c_master_transmit_receive(dev_handle, ...)` |
| 删除设备 | `i2c_driver_delete()` | `i2c_master_bus_rm_device()` + `i2c_del_master_bus()` |

### 关键区别

旧 API 用「总线号 + 设备地址」操作设备，新 API 用「设备句柄」操作设备。迁移步骤：

1. 用 `i2c_new_master_bus()` 创建总线，获得 `i2c_master_bus_handle_t`
2. 对每个 I2C 设备调用 `i2c_master_bus_add_device()`，获得 `i2c_master_dev_handle_t`
3. 所有读写操作使用设备句柄，不再传递地址

### 经验教训

- `i2c_master_transmit()` 和 `i2c_master_transmit_receive()` 的参数是**设备句柄**（`i2c_master_dev_handle_t`），不是总线句柄 + 地址。第一次迁移时搞混了，导致编译错误。
- 总线句柄需要保存为模块级静态变量，供触摸屏、音频编解码器等子模块使用。

## 二、esp_lcd I2C 面板驱动的 _Generic 陷阱

### 问题

ESP-IDF v5.5.1 的 `esp_lcd` 组件提供了两个版本的 I2C 面板驱动：
- `esp_lcd_panel_io_i2c_v1.c` — 使用旧 I2C API (`driver/i2c.h`)
- `esp_lcd_panel_io_i2c_v2.c` — 使用新 I2C API (`driver/i2c_master.h`)

头文件 `esp_lcd_io_i2c.h` 用 C11 `_Generic` 宏自动选择版本：

```c
#define esp_lcd_new_panel_io_i2c(bus, io_config, ret_io) _Generic((bus),  \
            i2c_master_bus_handle_t : esp_lcd_new_panel_io_i2c_v2, \
            default : esp_lcd_new_panel_io_i2c_v1) (bus, io_config, ret_io)
```

`esp_lcd_i2c_bus_handle_t` 被定义为 `uint32_t`。如果将 `i2c_master_bus_handle_t`（指针类型）强制转换为 `esp_lcd_i2c_bus_handle_t`：

```c
// 错误！_Generic 看到 uint32_t，匹配 default → v1（旧 API）
esp_lcd_new_panel_io_i2c((esp_lcd_i2c_bus_handle_t)i2c_bus_handle, ...);

// 正确！_Generic 看到 i2c_master_bus_handle_t → v2（新 API）
esp_lcd_new_panel_io_i2c(i2c_bus_handle, ...);
```

### 经验教训

- **不要盲目类型转换**。`esp_lcd_i2c_bus_handle_t` 看起来像个句柄类型，实际是 `uint32_t`。强制转换会丢失类型信息，导致 `_Generic` 匹配错误。
- 遇到 `_Generic` 相关的 API 时，确保传入的参数类型与期望的分支匹配。

## 三、IDF 旧 I2C 驱动的构造函数冲突检查

### 问题

IDF 的旧 I2C 驱动 (`components/driver/i2c/i2c.c`) 包含一个 `__attribute__((constructor))` 函数，在程序启动时（`do_global_ctors` 阶段，早于 `app_main`）自动运行：

```c
__attribute__((constructor))
static void check_i2c_driver_conflict(void)
{
    extern __attribute__((weak)) esp_err_t i2c_acquire_bus_handle(...);
    if ((void *)i2c_acquire_bus_handle != NULL) {
        ESP_EARLY_LOGE(I2C_TAG, "CONFLICT! driver_ng is not allowed to be used with this old driver");
        abort();
    }
}
```

它通过弱符号检测：如果新 I2C 驱动也被链接进二进制（即 `i2c_acquire_bus_handle` 有实现），就直接 abort。

### 为什么旧驱动会被链接

IDF 的 `esp_lcd` 组件的 CMakeLists.txt **同时编译** v1 和 v2：

```cmake
list(APPEND srcs "i2c/esp_lcd_panel_io_i2c_v1.c" "i2c/esp_lcd_panel_io_i2c_v2.c")
```

v1 包含 `driver/i2c.h` 并调用旧 I2C 函数，把旧驱动链入二进制。即使你只用 v2，旧驱动也在里面。

### 解决方案

在 `sdkconfig.defaults` 中添加：

```
CONFIG_I2C_SKIP_LEGACY_CONFLICT_CHECK=y
```

这会跳过构造函数中的冲突检查。因为 v1 的函数在运行时是死代码（没人调用），跳过检查是安全的。

### 经验教训

- `__attribute__((constructor))` 函数在 `app_main` 之前执行，报错的 backtrace 可能不包含 `app_main`，容易误导排查方向。
- 调试启动阶段崩溃时，注意看 backtrace 中的 `do_global_ctors` 帧。

## 四、sdkconfig.defaults 编码问题

### 问题

`sdkconfig.defaults` 中的中文注释导致构建失败：

```
UnicodeDecodeError: 'gbk' codec can't decode byte 0x80 in position XX
```

### 原因

IDF 的 Python 工具链（`kconfgen`）在中文 Windows 上使用系统默认编码（GBK）读取配置文件。但文件以 UTF-8 保存时，中文字符的 UTF-8 编码无法被 GBK 正确解码。

### 解决方案

sdkconfig.defaults 中的注释全部使用英文。

### 经验教训

- IDF 的 Python 脚本在中文 Windows 上有编码兼容问题。
- 不仅 sdkconfig.defaults，其他可能被 IDF Python 工具读取的配置文件也应注意编码。

## 五、第三方组件依赖导致 GCC ICE

### 问题

编译时报 GCC 内部错误：

```
internal compiler error: Segmentation fault
```

出错文件是 `dspi_conv_f32_ansi.c`（属于 `esp-dsp` 组件）。

### 排查过程

1. 错误发生在 `esp-dsp`，但项目代码并未直接使用 esp-dsp
2. 追溯依赖链：`esp-sr` → `esp-dsp`
3. `esp-sr`（语音识别库）在 `idf_component.yml` 中被声明为依赖，但项目代码中**没有任何 esp-sr 相关的 #include 或 API 调用**
4. 移除 `esp-sr` 依赖后，`esp-dsp` 不再被拉入，GCC ICE 消失

### 经验教训

- IDF 组件管理器会递归解析传递依赖。一个看似无害的依赖可能引入有 bug 的传递依赖。
- 遇到 GCC ICE 时，先确认出错的文件属于哪个组件，再追溯是谁引入的。
- 不要盲目添加"可能有用"的依赖，每个依赖都应确认被使用。

## 六、VS Code ESP-IDF 插件配置

### 问题

ESP-IDF 插件显示"配置无效"、无法选择芯片目标。

### 配置层级

插件的配置存储在三个地方，全部需要正确设置：

1. **项目级**: `.vscode/settings.json`
   - `idf.espIdfPathWin` / `idf.toolsPathWin`（Windows 用带 `Win` 后缀的 key）

2. **用户级**: `AppData/Roaming/Code/User/settings.json`
   - 全局默认 IDF 路径

3. **内部缓存**: `AppData/.../workspaceStorage/<hash>/state.vscdb`
   - SQLite 数据库，缓存 `idf.currentSetup` 等运行时状态
   - 即使修改了 settings.json，这里可能还缓存着旧路径
   - 需要用 sqlite3 命令直接修改：
     ```bash
     sqlite3 state.vscdb "UPDATE ItemTable SET value='...' WHERE key='idf.currentSetup';"
     ```

### 经验教训

- Windows 上必须用 `idf.espIdfPathWin`（不是 `idf.espIdfPath`）。
- 如果修改了所有 settings.json 还不生效，问题大概率在 state.vscdb 缓存。
- EIM（ESP-IDF Installation Manager）可能检测到错误的 Python（如 msys64 的），可以直接绕过 EIM，手动配置路径。

## 七、esp_codec_dev 的新旧 I2C 切换

### 机制

`esp_codec_dev` 组件的 I2C 控制层通过编译条件选择新旧 API：

```c
// audio_codec_ctrl_i2c.c
#if ESP_IDF_VERSION >= ESP_IDF_VERSION_VAL(5, 3, 0) && !CONFIG_CODEC_I2C_BACKWARD_COMPATIBLE
#include "driver/i2c_master.h"
#define USE_IDF_I2C_MASTER    // 使用新 API
#else
#include "driver/i2c.h"       // 使用旧 API
#endif
```

默认情况下（IDF >= 5.3 且未设置 backward compatible），自动使用新 API。

### 使用新 API 的配置方式

```c
// 旧 API 方式（用端口号）
audio_codec_i2c_cfg_t i2c_cfg = {
    .port = BSP_I2C_NUM,
    .addr = ES8311_CODEC_DEFAULT_ADDR,
};

// 新 API 方式（用总线句柄）
audio_codec_i2c_cfg_t i2c_cfg = {
    .bus_handle = i2c_bus_handle,
    .addr = ES8311_CODEC_DEFAULT_ADDR,
};
```

### 经验教训

- 迁移 I2C 时，不仅要改自己的代码，还要确认第三方组件是否也支持新 API，以及如何切换。
- 查看 managed_components 的 Kconfig 文件了解编译开关。

## 八、esp32-camera SCCB 驱动的版本选择

### 机制

`esp32-camera` 组件有两个 SCCB（I2C-like）驱动：

| 文件 | I2C API | 选择条件 |
|------|---------|---------|
| `driver/sccb.c` | 旧版 | IDF < 5.4 或 `CONFIG_SCCB_HARDWARE_I2C_DRIVER_LEGACY=y` |
| `driver/sccb-ng.c` | 新版 | IDF >= 5.4 且未设置 legacy（默认） |

CMakeLists.txt 中的选择逻辑：

```cmake
if (idf_version VERSION_GREATER_EQUAL "5.4" AND NOT CONFIG_SCCB_HARDWARE_I2C_DRIVER_LEGACY)
  list(APPEND srcs driver/sccb-ng.c)
else()
  list(APPEND srcs driver/sccb.c)
endif()
```

Kconfig 默认选择新驱动：

```
choice SCCB_HARDWARE_I2C_DRIVER_SELECTION
    default SCCB_HARDWARE_I2C_DRIVER_NEW
```

### 使用已初始化的 I2C 总线

```c
camera_config_t config = {
    .pin_sccb_sda = -1,        // -1 表示使用已初始化的 I2C 接口
    .pin_sccb_scl = CAMERA_PIN_SIOC,
    .sccb_i2c_port = 0,        // 使用 I2C 端口 0
};
```

`sccb-ng.c` 的 `SCCB_Use_Port()` 会通过 `i2c_master_get_bus_handle()` 获取已有总线句柄，不会重复初始化。

### 经验教训

- 第三方组件可能有新旧两套实现，需要确认编译的是哪一套。
- 查看 managed_components 的 CMakeLists.txt 和 Kconfig 了解编译条件。

## 九、Git 相关经验

### 子模块 vs 普通文件夹

`git submodule` 引用的仓库如果在本地有未推送的 commit，GitHub 上会显示为空链接（无法点击）。解决方案：

```bash
git submodule deinit -f Ref
git rm -f Ref
rm -rf .git/modules/Ref
# 删除 Ref/.git，作为普通文件夹重新添加
git add Ref/
```

### .gitignore

ESP-IDF 项目应排除以下生成文件：

```
build/
sdkconfig
sdkconfig.old
dependencies.lock
managed_components/
```

### LF/CRLF 警告

Windows 上 git 会提示 `LF will be replaced by CRLF`。这是正常的，不影响功能。可以在项目中添加 `.gitattributes` 统一换行符。

## 十、调试方法论

### 运行时崩溃排查

1. **看时间戳**：错误发生的毫秒数帮助判断是在启动阶段还是运行阶段
2. **看 backtrace**：从上到下是调用链的逆序，注意 `do_global_ctors` 表示全局构造函数阶段
3. **看 PC 地址**：配合 IDF 源码可以定位具体代码行

### 编译错误排查

1. 先确认出错文件属于哪个组件（自己的代码还是 managed_components）
2. 如果是第三方组件报错，检查 Kconfig 和 CMakeLists.txt 的编译条件
3. 如果是 GCC ICE，追溯传递依赖链

### 依赖冲突排查

1. `grep -r "i2c_driver_install\|i2c_param_config" managed_components/` — 找出谁在用旧 API
2. `grep "CONFIG_XXX" sdkconfig` — 确认编译开关的实际值
3. 查看 `build/config/sdkconfig.cmake` — 确认构建时的实际配置

## 十一、esp_board_manager CONFIG 选项与 gen_bmgr_codes

### 问题

`idf.py gen-bmgr-config` 生成的 `gen_bmgr_codes` 代码中 `#include "dev_audio_codec.h"` 等头文件编译报 "No such file or directory"。

### 原因

`esp_board_manager` 的设备/外设头文件位于子目录（如 `devices/dev_audio_codec/dev_audio_codec.h`），这些子目录只在对应 `CONFIG_ESP_BOARD_DEV_AUDIO_CODEC_SUPPORT=y` 启用时才通过 `INCLUDE_DIRS` 暴露。

当 sdkconfig 被重新生成（如修改 flash 大小导致 fullclean）时，这些 CONFIG 选项如果不在 sdkconfig.defaults 中声明，就会丢失为 "is not set"，导致头文件路径不可用。

### 解决方案

在 `sdkconfig.defaults` 中显式声明所有 SZP 板需要的 CONFIG 选项：

```
CONFIG_ESP_BOARD_PERIPH_I2C_SUPPORT=y
CONFIG_ESP_BOARD_PERIPH_I2S_SUPPORT=y
CONFIG_ESP_BOARD_PERIPH_SPI_SUPPORT=y
CONFIG_ESP_BOARD_PERIPH_LEDC_SUPPORT=y
CONFIG_ESP_BOARD_DEV_AUDIO_CODEC_SUPPORT=y
CONFIG_ESP_BOARD_DEV_DISPLAY_LCD_SUPPORT=y
CONFIG_ESP_BOARD_DEV_DISPLAY_LCD_SUB_SPI_SUPPORT=y
CONFIG_ESP_BOARD_DEV_LCD_TOUCH_I2C_SUPPORT=y
CONFIG_ESP_BOARD_DEV_CUSTOM_SUPPORT=y
```

### 经验教训

- `idf.py gen-bmgr-config` 生成的代码和 `esp_board_manager` 的编译选项有隐式依赖关系，但没有自动同步机制
- sdkconfig 容易在 reconfigure 时丢失选项，所有关键 CONFIG 都应写进 sdkconfig.defaults

## 十二、HTTP POST 中的 cJSON 悬空指针

### 问题

网页聊天发送中文消息时，LLM 回复 "I didn't quite understand that input — it looked like garbled text or random characters"。后续消息直接返回 "LLM error: invalid unicode code point"。

### 原因

```c
const char *message = msg_item->valuestring;  // 指向 cJSON 内部字符串
cJSON_Delete(root);                             // 释放整个 JSON 树
// ... message 现在是悬空指针
.user_text = message,                           // 使用了已释放的内存
```

`cJSON_Delete()` 释放了整个 JSON 树，包括 `valuestring` 指向的内存。之后的 `claw_core_submit()` 读取到的是已被覆盖的垃圾数据。

第一条消息（纯 ASCII "Hello"）可能碰巧没被覆盖所以偶尔成功，但中文（多字节 UTF-8）几乎必然被损坏。

### 解决方案

```c
char *message = strdup(msg_item->valuestring);  // 复制字符串
cJSON_Delete(root);                              // 安全释放 JSON 树
// ... 使用 message ...
free(message);                                   // 用完后释放副本
```

### 经验教训

- cJSON 的 `valuestring` 指针属于 cJSON 对象的内部内存，`cJSON_Delete()` 后失效
- 任何需要在 cJSON 树释放后继续使用字符串的场景，必须先 `strdup()`

## 十三、Telegram API 在国内被墙

### 问题

设备日志持续刷 Telegram 连接失败错误：

```
E esp-tls: [sock=51] delayed connect error: Software caused connection abort
E esp-tls: Failed to open new connection
W cap_im_tg: Telegram polling failed, retrying
```

### 原因

`api.telegram.org` 在中国大陆被 GFW 封锁，ESP32 的 TLS 连接直接被 reset。

### 解决方案

清除 NVS 中的 Telegram Bot Token，使 cap_im_tg 在启动时跳过：

```
W cap_im_tg: Telegram bot token not configured
W cap_im_tg: Telegram not configured, skipping start
```

清除方法：通过 Web 配网界面清空 Token 字段后保存。

### 经验教训

- 国内环境下 Telegram/QQ/飞书等 IM 通道各有限制，网页聊天是最可靠的无外部依赖交互方式
- esp-claw 的 NVS 设置在 `basic_demo` 命名空间下，可通过 HTTP API (`POST /api/config`) 修改

## 十四、Claude Code 中构建 ESP-IDF 项目

### 问题

从 Claude Code（Git Bash）调用 `idf.py` 时只输出 "MSys/Mingw is no longer supported" 后退出。

### 原因

`idf.py` 的 `__main__` 中检测到 `MSYSTEM` 环境变量后直接跳过 `main()` 调用：

```python
if 'MSYSTEM' in os.environ:
    print_warning('MSys/Mingw is no longer supported...')
elif ...:
    ...
else:
    main()  # 只有这里才会执行
```

Git Bash 会设置 `MSYSTEM` 环境变量，导致 idf.py 不执行任何命令。

### 解决方案

通过 `cmd.exe` 调用批处理文件，在批处理中 `set MSYSTEM=` 清除该变量：

```bat
@echo off
set MSYSTEM=
set IDF_PATH=e:\ESPIDF\551\v5.5.1\esp-idf
set IDF_TOOLS_PATH=e:\ESPIDF\551\Tools_551
set IDF_PYTHON_ENV_PATH=E:\ESPIDF\551\Tools_551\python_env\idf5.5_py3.11_env
set PYTHONIOENCODING=utf-8
set PATH=...cmake...ninja...xtensa-esp-elf...;%PATH%
python %IDF_PATH%\tools\idf.py build
```

此外需要 `set PYTHONIOENCODING=utf-8`，否则 idf.py 输出的 Unicode 字符（如 ✓）会在 GBK 终端上报 `UnicodeEncodeError`。

### 经验教训

- Git Bash 不是官方支持的 ESP-IDF 环境，需要绕过 MSYSTEM 检测
- Windows 中文环境下 Python 输出编码需要显式设置 UTF-8

## 十五、WiFi 扫描与热切换实现

### 背景

esp-claw 的 Web 配网界面只有手动输入 SSID/密码的方式，用户体验差。需要添加 WiFi 扫描、列表选择、无需重启的热切换功能。

### 实现架构

```
前端 (index.html + app.js + styles.css)
  ├── WiFi 扫描按钮 → GET /api/wifi/scan → 显示扫描结果弹窗
  ├── 点击列表项 → 自动填入 SSID → POST /api/wifi/connect → 热切换
  └── 保存配置 → 检测 SSID 变化 → 调用热切换而非重启

后端 (config_http_server.c + basic_demo_wifi.c/h)
  ├── wifi_scan_handler() → esp_wifi_scan_start() 阻塞扫描 → 返回 JSON
  └── wifi_connect_handler() → 处理 AP→APSTA 模式切换 + 断连重连
```

### 关键点

- WiFi 扫描必须在 APSTA 模式下执行（纯 AP 模式不支持扫描）
- 热切换流程：断开旧连接 → 设置新凭据 → 重新连接，期间保持 AP 热点不中断
- 前端通过 15 秒轮询检测连接结果，避免 WebSocket 等复杂实现

### 经验教训

- `esp_wifi_scan_start()` 的 `block=true` 参数在 APSTA 模式下会阻塞直到扫描完成，超时约 3-5 秒
- WiFi 断连重连期间，`basic_demo_wifi_event_handler()` 的重试计数器需要重置，否则可能触发错误的最大重试逻辑

## 十六、esp_video 组件依赖引入

### 问题

编译 camera DVP 驱动时报错：

```
fatal error: esp_video_init.h: No such file or directory
```

### 原因

`esp_board_manager` 的 camera DVP 驱动使用新的 `esp_video` 框架（v2.0.0+），而非旧的 `esp_camera` 驱动。该依赖在 `esp_board_manager` 的 `idf_component.yml` 中是条件依赖：

```yaml
espressif/esp_video:
  version: "^2.0.0"
  matches:
    - if: $CONFIG{ESP_BOARD_DEV_CAMERA_SUPPORT} == True
```

但 managed component 的依赖解析依赖 CMakeLists.txt 的 hash 校验。当 CMakeLists.txt 被修改（如添加 REQUIRES），hash 不匹配导致条件依赖不被解析。

### 解决方案

1. 在项目自身的 `main/idf_component.yml` 中显式声明依赖：

```yaml
espressif/esp_video:
  version: "^2.0.0"
  matches:
    - if: $CONFIG{ESP_BOARD_DEV_CAMERA_SUPPORT} == True
  public: true
```

2. 在 `esp_board_manager` 的 CMakeLists.txt REQUIRES 中添加 `espressif__esp_video` 和 `espressif__esp_cam_sensor`

3. 注意 `idf.py fullclean` 会删除 `managed_components/`，需要重新构建来下载 esp_video

### 经验教训

- managed component 的条件依赖依赖 CMakeLists.txt hash，修改后可能导致依赖不解析
- `esp_video` v2.0.0+ 和旧 `esp_camera` 是完全不同的框架，API 不兼容
- fullclean 会删除已下载的 managed components，后续构建需要重新下载

## 十七、CAP_LUA_MAX_MODULES 硬编码上限导致的 OOM 崩溃

### 问题

运行时崩溃：

```
E app_esp_claw: init_capabilities(270): Failed to register app Lua modules
ESP_ERROR_CHECK failed: esp_err_t 0x101 (ESP_ERR_NO_MEM)
```

### 排查过程

1. 错误码 `0x101` = `ESP_ERR_NO_MEM`，第一反应是内存不足
2. 检查运行日志发现 8 个硬件设备全部初始化成功，WiFi 正常，PSRAM 正常
3. 追踪到 `basic_demo_lua_modules_register()` → `cap_lua_register_module()`
4. 发现 `CAP_LUA_MAX_MODULES = 16`，而注册了 17 个模块

### 根因

`cap_lua_internal.h` 中：

```c
#define CAP_LUA_MAX_MODULES  16

static cap_lua_module_t s_modules[CAP_LUA_MAX_MODULES];  // 静态数组

esp_err_t cap_lua_register_module(const char *name, lua_CFunction open_fn)
{
    if (s_module_count >= CAP_LUA_MAX_MODULES) {
        return ESP_ERR_NO_MEM;  // 数组满了
    }
    ...
}
```

`basic_demo_lua_modules_register()` 注册了 17 个模块：delay/storage/gpio/led_strip/i2c/audio/button/display/board_manager/lcd_touch/camera/esp_heap/system/mcpwm/uart/adc/event_publisher。新增 camera 模块后刚好超限。

### 解决方案

将 `CAP_LUA_MAX_MODULES` 从 16 增大到 24。

### 经验教训

- `ESP_ERR_NO_MEM` 不一定是真正的内存不足，也可能是静态数组的硬编码上限
- 添加新功能模块时，应检查相关数组/缓冲区的容量限制
- esp-claw 的 `cap_lua` 组件是本地组件（`components/claw_capabilities/cap_lua`），可以直接修改

## 十八、cap_scheduler 初始化失败不应为致命错误

### 问题

设备不断重启循环，日志：
```
E cap_scheduler: Failed to load schedules from /fatfs/scheduler/schedules.json: ESP_ERR_INVALID_RESPONSE
ESP_ERROR_CHECK failed: esp_err_t 0x108 (ESP_ERR_INVALID_RESPONSE) at main.c:236
```

### 原因

`cap_scheduler_load_items()` 读取 schedules.json 后用 `cJSON_IsArray(root)` 校验，非数组内容返回 `ESP_ERR_INVALID_RESPONSE`。这个错误通过 `ESP_RETURN_ON_ERROR` → `ESP_ERROR_CHECK` 链传播到 `app_main`，触发 abort。

设备首次启动或 FAT 分区被格式化后，该文件可能不存在或内容无效。

### 解决方案

将 scheduler 初始化改为非致命：
```c
esp_err_t sched_err = cap_scheduler_init(&config);
if (sched_err != ESP_OK) {
    ESP_LOGW(TAG, "Scheduler init skipped (non-fatal): %s", esp_err_to_name(sched_err));
}
```

### 经验教训

- 对于非核心功能（定时调度、记忆系统等），初始化失败不应导致整个应用崩溃
- `ESP_ERROR_CHECK` 适合用在"必须成功否则系统无法工作"的场景，如硬件初始化
- `ESP_RETURN_ON_ERROR` 配合调用方的 `ESP_ERROR_CHECK` 会形成致命错误链，需要审慎使用

## 十九、PSRAM 栈与 flash cache 禁用的冲突

### 问题

时间同步成功后触发 assert：
```
assert failed: spi_flash_disable_interrupts_caches_and_other_cpu cache_utils.c:127
(esp_task_stack_is_sane_cache_disabled())
```

### 原因

ESP32-S3 的 Octal PSRAM 通过 SPI Flash 总线访问。当 NVS 写入等操作调用 `spi_flash_disable_interrupts_caches_and_other_cpu()` 时，flash cache 被禁用，PSRAM 同时变得不可访问。

IDF 在禁用 cache 前检查当前任务的栈指针是否指向内部 RAM。如果任务栈在 PSRAM 中，栈本身不可访问，IDF 直接 assert。

调用链：
```
cap_time_sync_service_task (PSRAM 栈)
  → cap_time_notify_sync_success
    → cap_scheduler_persist_runtime_state_locked
      → nvs_set_blob
        → spi_flash_disable_interrupts_caches_and_other_cpu  ← assert
```

### 解决方案

对可能调用 NVS/Flash 操作的任务，使用 `CLAW_TASK_STACK_INTERNAL_ONLY` 策略：
```c
claw_task_create(&(claw_task_config_t){
    .stack_size = 8192,
    .stack_policy = CLAW_TASK_STACK_INTERNAL_ONLY,
    ...
}, task_func, NULL, &handle);
```

### 经验教训

- `CLAW_TASK_STACK_PREFER_PSRAM` 会优先将栈分配到 PSRAM 以节省内部 RAM，但对涉及 Flash/NVS 操作的任务不安全
- ESP32-S3 的 PSRAM 和 Flash 共享 SPI 总线，禁用 flash cache 会同时影响 PSRAM
- 任何可能间接调用 `nvs_set_*` / `nvs_get_*` / `esp_partition_*` / `spi_flash_*` 的任务，其栈必须在内部 RAM
- 内部 RAM 紧张时可以减小栈大小，但不能放到 PSRAM

## 二十、PSRAM 栈问题的全局解决方案

### 问题

逐个任务排查 PSRAM 栈不安全的问题效率低且容易遗漏。每个涉及 Flash/NVS 的任务都需要单独修改栈策略。

### 原因

ESP32-S3 的 Octal PSRAM 与 Flash 共享 SPI 总线。`CONFIG_FREERTOS_TASK_CREATE_ALLOW_EXT_MEM=y` 允许 FreeRTOS 将任务栈分配到 PSRAM。任何间接调用 Flash 操作（NVS 写入、分区读取等）的任务，如果栈在 PSRAM 中，都会在 `spi_flash_disable_interrupts_caches_and_other_cpu()` 时触发 assert。

排查困难在于调用链很长且不可预测：看似无害的任务可能通过事件回调、状态持久化等间接调用 NVS。

### 解决方案

在 `sdkconfig.defaults` 和 `sdkconfig.defaults.board` 中全局禁用：

```
CONFIG_FREERTOS_TASK_CREATE_ALLOW_EXT_MEM=n
```

这使 `claw_task_external_memory_available()` 返回 false，所有 `CLAW_TASK_STACK_PREFER_PSRAM` 的任务自动回退到内部 RAM。无需逐个任务修改。

### 代价

内部 RAM 更紧张（每个任务栈占内部 RAM），但对于 210KB 可用 DRAM + 96KB DMA 预留的 ESP32-S3，扣除 WiFi 协议栈和 DMA 后仍可支撑 ~20 个任务。

### 经验教训

- PSRAM 栈问题具有传染性：一个任务使用 PSRAM 栈，其调用的任何函数如果间接操作 Flash，都会出问题
- 全局禁用是最安全的方案，避免逐个排查的遗漏风险
- 参考 m5stack_cores3 板级配置也是同样的做法

## 二十一、Display Arbiter — Emote 与 Lua 共享 LCD

### 背景

esp-claw 的 LCD 屏幕有两个使用者：emote 表情动画系统和 Lua 脚本显示。两者都通过 SPI 发送像素数据，不能同时操作。

### Display Arbiter 设计

`display_arbiter` 组件用引用计数管理 LCD 所有权：

- `DISPLAY_ARBITER_OWNER_EMOTE` — emote 动画系统（默认）
- `DISPLAY_ARBITER_OWNER_LUA` — Lua 脚本

Lua 通过 `display_arbiter_acquire(DISPLAY_ARBITER_OWNER_LUA)` 获取控制权，用完 `display_arbiter_release()` 释放。arbiter 在所有权变更时通过回调通知。

### 冲突解决

当 Lua 获取 LCD 时，emote 仍然持有 SPI panel IO 的 DMA 缓冲（~20KB 双缓冲）和 SPI 传输队列中的待处理事务。Lua 的 painter 初始化后第一次 SPI 发送会因 DMA 内存耗尽而失败。

解决方案（0426-0427 迭代）：

1. **emote_deinit** — 当 Lua 接管时，完全销毁 emote 实例，释放 DMA 缓冲和 SPI 队列
2. **NOP 回调替换** — emote_deinit 前将 SPI ISR 回调替换为 no-op，防止 use-after-free
3. **50ms 等待** — emote_deinit 后等待队列中残留事务完成
4. **max_transfer_sz 优化** — 从 12800 降到 3200，减少峰值 DMA 内存需求
5. **自动恢复** — Lua 释放后通过 reinit task 重建 emote，恢复 WiFi 状态
6. **可配置持续时间** — `display.set_resume_delay(seconds)` 控制显示时长，默认 5 秒

### 经验教训

- SPI 共享设备的冲突不仅是互斥问题，还涉及 DMA 缓冲和传输队列的释放
- `emote_deinit` 是同步的，但已提交的 SPI 事务在队列中是异步完成的，需要等待
- `gen_board_device_config.c` 是自动生成文件，修改 `trans_queue_depth` 后重新 `gen-bmgr-config` 会丢失

## 二十二、FreeRTOS Timer 任务栈溢出

### 问题

在 FreeRTOS Timer Service 任务（`Tmr Svc`）的回调中执行 `app_expression_emote_init()` 导致栈溢出崩溃：

```
***ERROR*** A stack overflow in task Tmr Svc has been detected.
Backtrace: panic_abort → vApplicationStackOverflowHook → vTaskSwitchContext
```

### 原因

FreeRTOS 的 `xTimerCreate()` 创建的软件定时器回调运行在 Timer Service 任务上下文中，该任务的默认栈很小（通常 2048 字节，由 `configTIMER_TASK_STACK_DEPTH` 控制）。

`app_expression_emote_init()` 执行了大量 I/O 操作：
- I2C 通信（PCA9557 CS 跳变）
- LCD 复位 + 初始化（ST7789）
- SPI 面板配置（mirror/swap/invert）
- emote 资源加载（分区 mmap）
- gfx 定时器创建

这些操作的栈需求远超 Timer 任务的栈大小。

### 解决方案

方案一（临时）：增大 `configTIMER_TASK_STACK_DEPTH`（不推荐，浪费且影响所有定时器）

方案二（最终）：用独立任务代替定时器回调：

```c
static void app_emote_reinit_task_func(void *arg)
{
    vTaskDelay(pdMS_TO_TICKS(3000));  // 延迟
    app_expression_emote_init();       // 重初始化（12KB 栈足够）
    s_emote_reinit_task = NULL;
    vTaskDelete(NULL);                 // 自行销毁
}
```

### 经验教训

- FreeRTOS Timer 回调**不适合**做任何 I/O 操作（I2C/SPI/UART/Flash），只适合设置标志位或释放信号量
- Timer Service 任务栈是全局共享资源，增大它会影响所有定时器的内存开销
- 需要延迟执行重活时，创建一次性任务（auto-delete task）是最干净的方案

## 二十三、SPI DMA 内存耗尽 — display 模块的核心陷阱

### 问题

Lua display 模块调用 `esp_lcd_panel_draw_bitmap` 发送像素数据时，SPI 报 `spi transmit (queue) color failed` (ESP_ERR_NO_MEM)，屏幕卡死无内容。

### 诊断过程

1. **错误假设 1**：`on_color_trans_done` 回调决定 SPI 使用 queue/polling 模式 → 尝试移除回调改 polling → 失败
2. **错误假设 2**：SPI 队列满是因为 emote 的异步 flush 通知堆积 → 尝试信号量同步 → 失败
3. **实际根因**：阅读 `esp_lcd_panel_io_spi.c` 源码发现 `panel_io_spi_tx_color` **始终**使用 `spi_device_queue_trans` 发送 color 数据（只有 command 阶段用 polling）

### 根因

```
display_hal 的 submit buffer 在 PSRAM (heap_caps_aligned_alloc MALLOC_CAP_SPIRAM)
  → esp_ptr_dma_capable() 返回 false
    → spi_master.c 的 setup_priv_desc() 分配 DMA 临时副本
      → 每个 chunk 分配 max_transfer_sz 大小的 DMA buffer
        → trans_queue_depth 个事务同时排队
          → 峰值 DMA = max_transfer_sz × queue_depth = 12800 × 10 = 128KB
            → 96KB DMA 池不够 → ESP_ERR_NO_MEM
```

### 解决方案

修改 `board_peripherals.yaml` 中 `max_transfer_sz` 从 12800 降到 3200：
- 峰值 DMA = 3200 × 2 = 6.4KB（gen-bmgr-config 自动将 queue_depth 从 10 降到 2）
- 代价：每次 SPI 传输的 chunk 更小，理论上刷新略慢，但 3200 bytes 对 320×16 像素行足够

### 关键认知

| 错误假设 | 实际情况 |
|---------|---------|
| `on_color_trans_done` 决定 queue/polling | color **始终**走 queue，只有 command 走 polling |
| PSRAM buffer 可以直接 DMA | `esp_ptr_dma_capable()` 只认内部 DRAM（0x3FCxxxxx），PSRAM 地址（0x3F4xxxxx）不算 DMA capable |
| SPI 队列满是并发问题 | 队列满是 DMA 内存池耗尽 |

### 经验教训

- ESP32-S3 的 PSRAM 虽然物理上支持 DMA（GDMA），但 IDF 的 `esp_ptr_dma_capable()` 只认内部 DRAM
- `max_transfer_sz` 和 `trans_queue_depth` 的乘积决定了峰值 DMA 内存需求
- 诊断 SPI 问题时应优先检查 DMA 内存状态，而非假设是并发/同步问题

## 二十四、Emote Reinit Task 自删除崩溃

### 问题

emote 恢复时创建 reinit task，如果 `app_expression_emote_init()` 失败，`app_emote_cleanup()` 尝试 `vTaskDelete(s_emote_reinit_task)` 删除自身 → FreeRTOS 状态损坏 → 崩溃。

### 原因

reinit task 调用链：
```
app_emote_reinit_task_func()
  → app_expression_emote_init() 失败
    → app_emote_cleanup()
      → vTaskDelete(s_emote_reinit_task)  // 删除自己！
```

FreeRTOS 的 `vTaskDelete()` 在删除当前任务时需要任务从 running 状态切换出去。在任务自己的执行流中删除自己，上下文切换过程中访问已释放的 TCB → 损坏。

### 解决方案

从 `app_emote_cleanup()` 中移除 `vTaskDelete(s_emote_reinit_task)`。reinit task 在自己的函数末尾通过 `vTaskDelete(NULL)` 自行退出：

```c
static void app_emote_cleanup(void)
{
    /* Do NOT delete s_emote_reinit_task here — if called from
     * within the reinit task itself, self-deletion corrupts FreeRTOS state. */
    if (s_emote_handle) {
        emote_deinit(s_emote_handle);
        s_emote_handle = NULL;
    }
    display_arbiter_set_owner_changed_callback(NULL, NULL);
}
```

### 经验教训

- `vTaskDelete(task)` 中 task 指向当前执行的任务时，等价于 `vTaskDelete(NULL)`，但 FreeRTOS 对此处理存在时序陷阱
- 清理函数不应假设自己运行在哪个任务上下文中

## 二十五、构建系统 managed component 依赖修复

### 问题

删除 build 目录重新 cmake 后，`esp_board_manager` 的设备子模块编译报头文件找不到（`esp_video_init.h`、`esp_codec_dev.h`、`esp_lcd_touch.h`、`dev_ledc_ctrl.h`）。

### 原因

多层问题叠加：
1. `esp_board_manager` 的 `idf_component.yml` 条件依赖需要 CMakeLists.txt hash 匹配才能解析。hash 不匹配时 `espressif__esp_video` 等不加入 include path
2. `gen_bmgr_codes` 的 REQUIRES 引用 `esp_board_manager`（短名），但组件注册名为 `espressif__esp_board_manager`，include path 未传播
3. `gen_board_device_config.c` 引用 `dev_ledc_ctrl.h`，但 `CONFIG_ESP_BOARD_DEV_LEDC_CTRL_SUPPORT` 未启用

### 解决方案

1. `esp_board_manager` CMakeLists.txt：手动添加 `espressif__esp_codec_dev`、`espressif__esp_lcd_touch`、`espressif__esp_video` 到 REQUIRES
2. `gen_bmgr_codes` CMakeLists.txt：添加所有设备子目录到 INCLUDE_DIRS
3. `sdkconfig.defaults`：添加 `CONFIG_ESP_BOARD_DEV_LEDC_CTRL_SUPPORT=y`

### 经验教训

- managed component 的条件依赖机制不可靠（依赖 hash），关键依赖应手动声明
- `gen-bmgr-config` 生成的代码可能引用未启用 CONFIG 的设备，需要在 defaults 中补全
- `idf.py fullclean` 或删除 build 目录后，managed component 依赖需要重新解析
