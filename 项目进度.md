# ESP-CLAW 移植项目进度

## 项目概述

将 [esp-claw](https://github.com/espressif/esp-claw)（乐鑫 AI Agent 框架）移植到**立创实战派 ESP32-S3 开发板**。

- **GitHub**: https://github.com/GBall5599/-ESP-CLAW
- **ESP-IDF**: v5.5.1
- **开发板**: ESP32-S3-WROOM-1-N16R8 (16MB Flash + 8MB PSRAM)
- **LLM 后端**: DeepSeek-v4-flash（OpenAI Compatible API）

原始例程基于 IDF v5.4.3，已完成以下迁移工作：

##### I2C 驱动迁移（核心改动）
旧版 I2C 驱动 (`driver/i2c.h`) 与新版 (`driver/i2c_master.h`) 不能在同一 I2C 端口混用。涉及改动：

- **`esp32_s3_szp.h`**: `#include "driver/i2c.h"` → `#include "driver/i2c_master.h"`
- **`esp32_s3_szp.c` - `bsp_i2c_init()`**: 用 `i2c_new_master_bus()` 创建总线，再用 `i2c_master_bus_add_device()` 为 QMI8658、PCA9557 创建设备句柄
- **QMI8658/PCA9557 读写**: `i2c_master_write_read_device()` / `i2c_master_write_to_device()` → `i2c_master_transmit_receive()` / `i2c_master_transmit()`（使用设备句柄）
- **触摸屏初始化**: 去掉 `(esp_lcd_i2c_bus_handle_t)` 类型转换，让 C11 `_Generic` 宏正确选择 v2 I2C 驱动；添加 `scl_speed_hz = 100000`
- **音频编解码器**: ES8311/ES7210 的 I2C 配置从 `.port = BSP_I2C_NUM` 改为 `.bus_handle = i2c_bus_handle`

##### sdkconfig.defaults 适配
- 中文注释替换为英文（IDF 的 kconfgen 使用 GBK 编码，无法读取 UTF-8 中文）
- 添加 `CONFIG_I2C_SKIP_LEGACY_CONFLICT_CHECK=y`（IDF 的 esp_lcd 组件同时编译 v1/v2 I2C 驱动，需跳过冲突检查）

##### 依赖清理
- 从 `idf_component.yml` 移除 `espressif/esp-sr: "~1.6.0"`（项目未使用语音识别功能，esp-sr 的传递依赖 esp-dsp 会触发 GCC 内部编译器错误 Segmentation Fault）

## 踩坑记录

### 1. VS Code ESP-IDF 插件配置

插件存储配置在三个层级，全部需要正确配置才能工作：
- **workspace settings**: `.vscode/settings.json`（项目级）
- **global settings**: `AppData/Roaming/Code/User/settings.json`（用户级）
- **内部缓存**: `AppData/Roaming/Code/User/workspaceStorage/<hash>/state.vscdb`（SQLite 数据库）

其中 state.vscdb 的 `idf.currentSetup` 字段会缓存旧路径，导致芯片目标选择功能异常。需要用 sqlite3 直接修改数据库才能修复。

### 2. sdkconfig.defaults 编码问题

IDF 的 Python 工具链（kconfgen）在中文 Windows 上使用 GBK 编码读取配置文件。UTF-8 中文注释会导致 `UnicodeDecodeError: 'gbk' codec can't decode byte`。解决方案：注释全部使用英文。

### 3. I2C 新旧驱动冲突

ESP-IDF v5.5.x 中，旧 I2C 驱动 (`driver/i2c.h`) 和新 I2C 驱动 (`driver/i2c_master.h`) 不能在同一端口混用。旧驱动源码中有 `__attribute__((constructor))` 函数，在程序启动时自动检测并 abort。

此外，IDF 的 `esp_lcd` 组件总是同时编译 v1（旧 API）和 v2（新 API）两个 LCD I2C 面板驱动。即使只用 v2，v1 也会把旧 I2C 驱动链入二进制。需要设置 `CONFIG_I2C_SKIP_LEGACY_CONFLICT_CHECK=y` 跳过检查。

### 4. _Generic 宏类型匹配

`esp_lcd_new_panel_io_i2c()` 在 C 中用 `_Generic` 宏根据 bus 参数类型选择 v1 或 v2。如果将 `i2c_master_bus_handle_t`（指针类型）强制转换为 `esp_lcd_i2c_bus_handle_t`（= `uint32_t`），`_Generic` 会匹配到 `default` 分支即 v1（旧 API），导致运行时 I2C 冲突。正确做法是直接传 `i2c_master_bus_handle_t` 类型的变量。

### 5. esp-dsp GCC 内部编译器错误

`espressif/esp-sr` 组件的传递依赖 `esp-dsp` 在编译 `dspi_conv_f32_ansi.c` 时触发 GCC ICE（Internal Compiler Error: Segmentation Fault）。esp-sr 在本项目中未使用，移除后编译通过。

### 6. gen_bmgr_codes CONFIG 选项丢失

`idf.py gen-bmgr-config` 生成的代码引用了 `esp_board_manager` 设备/外设子目录的头文件（如 `dev_audio_codec.h`），但这些头文件的路径只在对应的 `CONFIG_ESP_BOARD_*_SUPPORT` 选项启用时才会被加入 INCLUDE_DIRS。如果 sdkconfig 被重新生成导致这些选项丢失，编译会报 "No such file or directory"。

解决方案：在 `sdkconfig.defaults` 中显式声明所有需要的 CONFIG 选项。

### 7. HTTP POST 处理中的 cJSON 悬空指针

`chat_ask_handler()` 中通过 `cJSON_GetObjectItemCaseSensitive()` 获取的字符串指针在 `cJSON_Delete()` 后变成悬空指针，导致发给 LLM 的中文消息变成乱码。

解决方案：用 `strdup()` 复制字符串，在 `claw_core_submit()` 后 `free()` 释放副本。

### 8. Telegram API 被墙

Telegram Bot API (`api.telegram.org`) 在中国大陆无法直接访问，导致 `esp-tls: Failed to open new connection` 循环刷错误日志。清除 NVS 中的 tg_bot_token 后恢复正常。

### 9. claw_core 超时参数双重单位转换

`claw_core_submit()` 和 `claw_core_receive_for()` 的 timeout 参数已经是**毫秒**单位，但代码中误用了 `pdMS_TO_TICKS()` 做了一次转换（将 ms → ticks），函数内部又做了一次转换（ticks → ticks），导致实际超时时间被缩短约 10 倍。

例如 `pdMS_TO_TICKS(120000)` 在 1000Hz tick 下变成 120000 ticks，函数内部再 `pdMS_TO_TICKS(120000)` = 120000 ticks = 120 秒。但实际上 FreeRTOS tick rate 不是 1000Hz 而是 100Hz，所以 120000 ms 被转成 12000 ticks，函数内部再按 ms 处理时实际变成了 `pdMS_TO_TICKS(12000)` = 1200 ticks = 12 秒。

解决方案：直接传毫秒值，不做 `pdMS_TO_TICKS` 转换。

### 10. esp_video 组件依赖引入

`esp_board_manager` 的 camera DVP 驱动依赖 `esp_video` 框架（v2.0.0+），但该依赖在 `esp_board_manager` 的 `idf_component.yml` 中是条件依赖，需要通过 CMakeLists.txt 的 hash 匹配才能正确解析。

当修改了 `esp_board_manager` 的 CMakeLists.txt 后（如添加 REQUIRES），managed component 的 hash 校验会失败，导致依赖解析中断。解决方案是在项目自身的 `main/idf_component.yml` 中显式声明 `espressif/esp_video: ^2.0.0` 依赖。

### 11. CAP_LUA_MAX_MODULES 硬编码上限

`cap_lua` 组件中 `CAP_LUA_MAX_MODULES` 定义为 16，限制了可注册的 Lua 模块数量。新增 camera Lua 模块后，总模块数达到 17 个，超出上限导致 `cap_lua_register_module()` 返回 `ESP_ERR_NO_MEM`，表现为运行时崩溃。

错误日志 `ESP_ERROR_CHECK failed: esp_err_t 0x101 (ESP_ERR_NO_MEM)` 容易被误判为内存不足，实际是数组越界保护。解决方案：将 `CAP_LUA_MAX_MODULES` 从 16 增大到 24。

### 12. scheduler 初始化失败导致重启循环

`/fatfs/scheduler/schedules.json` 文件内容无效（非 JSON 数组），`cap_scheduler_load_items()` 返回 `ESP_ERR_INVALID_RESPONSE`。该错误通过 `ESP_RETURN_ON_ERROR` → `ESP_ERROR_CHECK` 链触发 abort，导致设备不断重启。

修复：将 `app_claw.c` 中 scheduler 初始化从 `ESP_RETURN_ON_ERROR` 改为 `ESP_LOGW` 非致命处理。

### 13. PSRAM 栈在 flash cache 禁用时的陷阱

ESP32-S3 的 PSRAM 通过 SPI Flash 总线访问。当 NVS 写入等操作需要临时禁用 flash cache 时，PSRAM 中的数据（包括任务栈）变得不可访问。IDF 在 `spi_flash_disable_interrupts_caches_and_other_cpu()` 中检查当前任务栈是否在内部 RAM，如果不在则触发 assert。

`CLAW_TASK_STACK_PREFER_PSRAM` 会优先将任务栈分配到 PSRAM，对于会调用 NVS/Flash 操作的任务来说是不安全的。修复：使用 `CLAW_TASK_STACK_INTERNAL_ONLY` 并适当增大栈大小。
