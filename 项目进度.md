# ESP-CLAW 移植项目进度

## 项目概述

将 [esp-claw](https://github.com/espressif/esp-claw)（乐鑫 AI Agent 框架）移植到**立创实战派 ESP32-S3 开发板**。

- **GitHub**: https://github.com/GBall5599/-ESP-CLAW
- **ESP-IDF**: v5.5.1
- **开发板**: ESP32-S3-WROOM-1-N16R8 (16MB Flash + 8MB PSRAM)
- **LLM 后端**: DeepSeek-v4-flash（OpenAI Compatible API）

#### 3. 例程 IDF v5.5.1 适配 (`14-handheld(new)/`)

原始例程基于 IDF v5.4.3，已完成以下迁移工作：

##### I2C 驱动迁移（核心改动）
旧版 I2C 驱动 (`driver/i2c.h`) 与新版 (`driver/i2c_master.h`) 不能在同一 I2C 端口混用。涉及改动：

- **`esp32_s3_szp.h`**: `#include "driver/i2c.h"` → `#include "driver/i2c_master.h"`
- **`esp32_s3_szp.c` - `bsp_i2c_init()`**: 用 `i2c_new_master_bus()` 创建总线，再用 `i2c_master_bus_add_device()` 为 QMI8658、PCA9557 创建设备句柄
- **QMI8658/PCA9557 读写**: `i2c_master_write_read_device()` / `i2c_master_write_to_device()` → `i2c_master_transmit_receive()` / `i2c_master_transmit()`（使用设备句柄）
- **触摸屏初始化**: 去掉 `(esp_lcd_i2c_bus_handle_t)` 类型转换，让 C11 `_Generic` 宏正确选择 v2 I2C 驱动；添加 `scl_speed_hz = 100000`
- **音频编解码器**: ES8311/ES7210 的 I2C 配置从 `.port = BSP_I2C_NUM` 改为 `.bus_handle = i2c_bus_handle`

##### sdkconfig.defaults 适配
- 中文注释替换为英文（IDF 的 kconfgen 使用 GBK 编码，无法读取 UTF-8 中文）
- 添加 `CONFIG_I2C_SKIP_LEGACY_CONFLICT_CHECK=y`（IDF 的 esp_lcd 组件同时编译 v1/v2 I2C 驱动，需跳过冲突检查）

##### 依赖清理
- 从 `idf_component.yml` 移除 `espressif/esp-sr: "~1.6.0"`（项目未使用语音识别功能，esp-sr 的传递依赖 esp-dsp 会触发 GCC 内部编译器错误 Segmentation Fault）

## 踩坑记录

### 1. VS Code ESP-IDF 插件配置

插件存储配置在三个层级，全部需要正确配置才能工作：
- **workspace settings**: `.vscode/settings.json`（项目级）
- **global settings**: `AppData/Roaming/Code/User/settings.json`（用户级）
- **内部缓存**: `AppData/.../workspaceStorage/<hash>/state.vscdb`（SQLite 数据库）

其中 state.vscdb 的 `idf.currentSetup` 字段会缓存旧路径。需要用 sqlite3 直接修改数据库。

### 2. sdkconfig.defaults 编码问题

IDF 的 Python 工具链（kconfgen）在中文 Windows 上使用 GBK 编码。UTF-8 中文注释导致构建失败。注释全部使用英文。

### 3. I2C 新旧驱动冲突

旧 I2C 驱动有 `__attribute__((constructor))` 冲突检查，`esp_lcd` 同时编译 v1/v2 会链入旧驱动。设置 `CONFIG_I2C_SKIP_LEGACY_CONFLICT_CHECK=y`。

### 4. _Generic 宏类型匹配

不要将 `i2c_master_bus_handle_t` 强转为 `esp_lcd_i2c_bus_handle_t`（= uint32_t），否则 `_Generic` 匹配错误。

### 5. GCC ICE（非确定性）

IDF v5.5.1 + xtensa-gcc 14.2.0 编译 `esp_lcd_panel_rgb.c` 时偶尔触发 Segmentation Fault。重试即可。

### 6. esp_board_manager 条件依赖不解析

managed component 的 `idf_component.yml` 条件依赖需要 CMakeLists hash 匹配。修改 CMakeLists 后 hash 失败，导致 `espressif__esp_video`/`espressif__esp_codec_dev` 等不加入 include path。解决方案：手动添加 REQUIRES + include paths。

### 7. SPI DMA 内存耗尽（核心教训）

`panel_io_spi_tx_color` **始终**使用 `spi_device_queue_trans` 发送 color 数据，无论是否注册了 `on_color_trans_done` 回调。当 color buffer 在 PSRAM 时，`setup_priv_desc` 自动分配 DMA 副本。峰值 DMA = max_transfer_sz × trans_queue_depth = 12800 × 10 = 128KB > 96KB DMA 池 → ESP_ERR_NO_MEM。

修正了之前错误的假设：以为 `on_color_trans_done` 回调决定了 SPI 使用 queue 还是 polling 模式。实际上 polling 只用于 command 阶段，color 数据始终走 queue。

### 8. PSRAM 栈与 flash cache 的冲突

ESP32-S3 Octal PSRAM 通过 SPI Flash 总线访问。禁用 flash cache 时 PSRAM 不可访问。全局禁用 PSRAM 栈 `CONFIG_FREERTOS_TASK_CREATE_ALLOW_EXT_MEM=n`。

### 9. FreeRTOS Timer 回调不能做 I/O

Timer Service 任务栈很小（~2KB）。回调中不能做 I2C/SPI/Flash 操作，应使用独立 task。
