## 问题 1：LCD 显示屏不亮（已解决）

### 现象
ST7789 LCD 屏幕背光亮，但无内容显示。

### 根本原因
PCA9557 控制 LCD_CS 永久为 LOW。board_manager 初始化时拉低 CS，但 ST7789 软件复位后需要 CS 跳变才能正确同步 SPI。参考工程（14-handheld）在每次 SPI 操作前后手动切换 CS，而 board_manager 的 SPI 驱动不知道 CS 是由 I2C 扩展器控制的。

### 解决方案（2026-04-25, commit f4d1d97）
在 `app_expression_emote_init()` 中，board_manager 初始化完成后手动执行 CS 跳变：
1. 通过 I2C 向 PCA9557 发送 CS=HIGH（0x03）
2. 等待 10ms
3. 通过 I2C 向 PCA9557 发送 CS=LOW（0x02）
4. 等待 10ms
5. 执行 `esp_lcd_panel_reset()` + 150ms 延迟
6. 重新 `esp_lcd_panel_init()` + mirror/swap/invert 配置

仅首次启动执行（`s_lcd_needs_reinit` 标志），Lua 释放后 emote 恢复时跳过。

### 验证
烧录后 LCD 正常显示 emote 表情 + WiFi 状态文字。

## 问题 2：设备不断崩溃重启（已解决）

### 崩溃 A：scheduler 初始化失败
**原因**: `/fatfs/scheduler/schedules.json` 内容无效，`ESP_ERROR_CHECK` 链触发 abort。
**修复**: 改为非致命 `ESP_LOGW`（2026-04-25, commit 09c0366）。

### 崩溃 B：cap_time 栈溢出
**原因**: 任务栈在 PSRAM，NVS 写入时 flash cache 禁用导致 assert。
**修复**: 栈策略改为 INTERNAL_ONLY + 8KB（2026-04-25, commit 2b7eb43）。

### 崩溃 C：QQ 远程指令触发重启
**原因**: 多个 claw_task 使用 PSRAM 栈，flash cache assert。
**修复**: 全局禁用 PSRAM 栈 `CONFIG_FREERTOS_TASK_CREATE_ALLOW_EXT_MEM=n`（2026-04-26, commit 2b7eb43）。

## 问题 3：LCD 无法显示 Lua 绘制内容（已解决）

### 现象
Agent 通过 Lua display 模块绘制内容（Hello World、图片），但 SPI 发送失败，屏幕显示 emote 内容不变。

### 排查过程
最初假设 emote 注册了 `on_color_trans_done` 回调导致 SPI 使用异步模式。两次修复尝试（信号量方式、polling 模式）均失败。

### 根本原因（2026-04-27 确认）
**SPI DMA 内存耗尽**。`panel_io_spi_tx_color` **始终**使用 `spi_device_queue_trans` 发送 color 数据（无论是否注册回调）。当 color buffer 在 PSRAM（非 DMA capable）时，`spi_master.c` 的 `setup_priv_desc` 自动分配 DMA 副本。

峰值 DMA 内存 = max_transfer_sz × trans_queue_depth：
- 修改前: 12800 × 10 = 128KB > 96KB DMA 池 → ESP_ERR_NO_MEM
- 修改后: 3200 × 2 = 6.4KB < 96KB → 正常

### 解决方案（2026-04-27，待烧录验证）

1. **`board_peripherals.yaml`**: `max_transfer_sz` 从 12800 降到 3200
2. **`gen_board_device_config.c`**: `trans_queue_depth` 自动从 10 降到 2
3. **`app_expression_emote.c`**:
   - NOP 回调替换（防止 Lua 接管后 ISR 调用已释放的 emote 回调）
   - 50ms 等待（让队列中残留事务完成）
4. **Reinit task 自删除修复**: `app_emote_cleanup()` 不再删除 reinit task
5. **WiFi 状态保存/恢复**: Lua 释放后 emote 恢复正确状态
6. **可配置显示持续时间**: `display.set_resume_delay(seconds)` API

### 修正的错误认知
| 错误假设 | 实际情况 |
|---------|---------|
| `on_color_trans_done` 决定 SPI queue/polling 模式 | color 数据**始终**走 queue，只有 command 走 polling |
| PSRAM buffer 可以直接 DMA | `esp_ptr_dma_capable()` 只认内部 DRAM（0x3FCxxxxx） |
| SPI 队列满是因为 emote 的异步回调 | 队列满是因为 DMA 内存池耗尽 |

## 问题 4：构建系统依赖解析失败（已解决）

### 现象
删除 build 目录重新 cmake 后，编译报 `dev_camera.h`/`dev_audio_codec.h`/`dev_lcd_touch_i2c.h` 等找不到头文件。

### 根本原因
1. `esp_board_manager` 的 `idf_component.yml` 声明了条件依赖（`espressif__esp_video` 等），但 managed component CMakeLists hash 校验失败导致依赖不解析
2. `gen_bmgr_codes` 的 REQUIRES 引用 `esp_board_manager`（短名），但组件注册名为 `espressif__esp_board_manager`
3. `gen_board_device_config.c` 引用 `dev_ledc_ctrl.h`，但 `CONFIG_ESP_BOARD_DEV_LEDC_CTRL_SUPPORT` 未启用

### 解决方案
1. 在 `esp_board_manager` CMakeLists.txt 的 REQUIRES 中手动添加 `espressif__esp_codec_dev`、`espressif__esp_lcd_touch`、`espressif__esp_video`
2. 在 `gen_bmgr_codes` CMakeLists.txt 中显式添加所有设备子目录的 INCLUDE_DIRS
3. 在 sdkconfig.defaults 中添加 `CONFIG_ESP_BOARD_DEV_LEDC_CTRL_SUPPORT=y`

### 经验教训
- managed component 的条件依赖机制在 hash 校验失败时不工作
- `gen-bmgr-config` 生成的代码可能引用尚未启用 CONFIG 的设备头文件
- 全量 clean build 时必须确保所有 CONFIG 选项在 sdkconfig.defaults 中声明

## 问题 5：GCC ICE（Internal Compiler Error）

### 现象
编译 `esp_lcd_panel_rgb.c` 时偶发 `internal compiler error: Segmentation fault`。

### 原因
IDF v5.5.1 + xtensa-esp-elf-gcc 14.2.0 的非确定性 bug。同一源文件在同一编译选项下，有时 ICE 有时不 ICE。

### 解决方案
重试编译即可。ninja 会从失败处继续，不需要 fullclean。
