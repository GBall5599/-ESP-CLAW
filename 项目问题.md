# 项目问题记录

## 问题 1：LCD 显示屏不亮（排查中）

### 现象
ST7789 LCD 屏幕背光亮，但无内容显示。同一个开发板运行 14-handheld 参考工程时 LCD 正常。

### 排查过程

#### 假设 1：XIP (Execute In Place from PSRAM) 干扰 SPI
- **猜测**：`CONFIG_SPIRAM_XIP_FROM_PSRAM=y` 将代码放到 PSRAM 执行，可能影响 SPI DMA
- **尝试**：在 `sdkconfig.defaults` 和 `sdkconfig.defaults.board` 中设置 `CONFIG_SPIRAM_XIP_FROM_PSRAM=n`
- **结果**：无效，LCD 仍不亮
- **备注**：IDF 的 `idf.py set-target` 会覆盖 defaults 文件中的 XIP 设置，需要手动编辑 sdkconfig

#### 假设 2：背光未打开
- **猜测**：LEDC PWM 初始化时 duty=0，背光可能没开
- **尝试**：在 `board_devices.yaml` 添加 `ledc_ctrl` 设备（`default_percent: 100`）
- **结果**：背光确实变亮了（duty 从 0 变为 1023），但 LCD 仍无内容
- **备注**：用户反馈之前就能看到背光，说明这不是根本原因

#### 假设 3：SPI 总线被其他模块预先占用
- **猜测**：`periph_spi.c` 接受 `ESP_ERR_INVALID_STATE`，如果总线已被占用，引脚配置可能被忽略
- **尝试**：在 `periph_spi.c` 添加日志打印 `spi_bus_initialize` 的实际返回值
- **结果**：返回 `ESP_OK`，总线未被占用

#### 假设 4：PCA9557 LCD_CS 未拉低
- **猜测**：I2C 写入 PCA9557 可能静默失败，CS 实际为 HIGH
- **尝试**：在 `setup_device.c` 添加 PCA9557 输出寄存器回读
- **结果**：`OUTPUT=0x02 (LCD_CS=0, PA_EN=1, DVP_PWDN=0)`，CS 确实为 LOW

#### 假设 5：esp_lcd 面板 IO 层有问题
- **猜测**：esp_lcd 框架的 SPI 面板 IO 可能发送失败但返回 OK
- **尝试**：绕过 esp_lcd，直接用 `spi_device_polling_transmit` 发送 ST7789 命令
- **结果**：所有 SPI 事务返回 ESP_OK，但屏幕仍无内容。原因是 raw SPI 设备与 esp_lcd 已有的 SPI 设备在同一总线上冲突

#### 假设 6：崩溃重启导致 emote 来不及渲染
- **猜测**：设备不断重启，emote 系统无法完成首帧渲染
- **尝试**：修复崩溃问题（见问题 2）后观察
- **结果**：设备不再重启，系统稳定运行，但 LCD 仍然只亮背光、无内容
- **备注**：emote 系统初始化成功（日志显示 "Expression emote app initialized"），assets 加载正常（2 emoji, 1 icon, 4 layout），display arbiter 默认授予 EMOTE 所有权

#### 假设 7：emote flush 回调未被调用
- **猜测**：emote 渲染管线可能未触发 flush，导致数据未发送到 LCD
- **尝试**：在 `app_emote_flush_callback` 中添加计数器日志，记录 flush 调用次数和 `esp_lcd_panel_draw_bitmap` 返回值
- **状态**：已添加诊断代码，待用户烧录测试验证

### 已确认正常的环节
- SPI 总线初始化（SPI3_HOST，80MHz）
- PCA9557 CS 拉低
- `esp_lcd_panel_init()` 成功（SLPOUT + MADCTL + COLMOD + RAMCTRL + DISPON 全部返回 ESP_OK）
- `esp_lcd_panel_mirror/swap_xy/invert_color` 全部返回 ESP_OK
- emote 系统初始化成功
- display arbiter 默认 EMOTE 拥有所有权

### 待确认
- emote flush 回调是否被调用
- `esp_lcd_panel_draw_bitmap` 是否正常工作
- 像素数据是否正确到达 ST7789

### 14-handheld 参考对比
两项目的 LCD 配置完全一致：SPI3_HOST、80MHz、spi_mode=2、无硬件复位引脚（`reset_gpio_num=-1`）、同样使用 `esp_lcd_new_panel_st7789`。参考工程正常显示，esp-claw 不显示，差异不在硬件配置层面。

## 问题 2：设备不断崩溃重启（已解决）

这个问题实际包含两个独立的崩溃，按发生顺序记录。

### 崩溃 A：scheduler 初始化失败导致 abort（首要问题）

#### 现象
设备在 Wi-Fi 连接成功、`app_claw_start()` 阶段崩溃重启，日志：
```
E cap_scheduler: Failed to load schedules from /fatfs/scheduler/schedules.json: ESP_ERR_INVALID_RESPONSE
E cap_scheduler: Scheduler init load failed: ESP_ERR_INVALID_RESPONSE
E app_esp_claw: app_claw_start(346): Failed to init scheduler
ESP_ERROR_CHECK failed: esp_err_t 0x108 (ESP_ERR_INVALID_RESPONSE) at main.c:236
```

#### 根本原因
`cap_scheduler_load_items()` 从 FAT 分区读取 `/fatfs/scheduler/schedules.json`，文件内容不是有效的 JSON 数组（`cJSON_IsArray(root)` 返回 false），返回 `ESP_ERR_INVALID_RESPONSE`。该错误通过 `ESP_RETURN_ON_ERROR` 一路传播到 `app_main` 的 `ESP_ERROR_CHECK`，触发 abort。

#### 修复方案
将 `app_claw.c` 中 scheduler 初始化从致命错误改为非致命：
```c
// 修改前
ESP_RETURN_ON_ERROR(cap_scheduler_init(&config), TAG, "Failed to init scheduler");

// 修改后
esp_err_t sched_err = cap_scheduler_init(&config);
if (sched_err != ESP_OK) {
    ESP_LOGW(TAG, "Scheduler init skipped (non-fatal): %s", esp_err_to_name(sched_err));
}
```

#### 文件
`esp-claw/application/basic_demo/main/app_claw.c` 约第 346 行

### 崩溃 B：cap_time 任务栈溢出导致 assert（潜在问题）

#### 现象
如果崩溃 A 被绕过（scheduler 初始化成功），系统进入时间同步阶段后可能触发：
```
assert failed: spi_flash_disable_interrupts_caches_and_other_cpu cache_utils.c:127
(esp_task_stack_is_sane_cache_disabled())
```

#### 崩溃调用栈
```
cap_time_sync_service_task (栈空间不足)
  → cap_time_notify_sync_success
    → basic_demo_time_sync_success
      → cap_scheduler_handle_time_sync
        → cap_scheduler_persist_runtime_state_locked
          → nvs_set_blob
            → esp_partition_read
              → spi_flash_disable_interrupts_caches_and_other_cpu ← 栈溢出
```

#### 根本原因
`cap_time_sync_service_task` 使用 `CLAW_TASK_STACK_PREFER_PSRAM` 策略分配栈到 PSRAM。当 NVS 写入需要临时禁用 flash cache 时，PSRAM 变为不可访问（PSRAM 也通过 SPI Flash 访问），IDF 检查当前任务栈是否在内部 RAM，发现不在，触发 assert。

#### 修复方案
修改 `cap_time.c` 中任务创建参数：
```c
// 修改前
.stack_size = 4096,
.stack_policy = CLAW_TASK_STACK_PREFER_PSRAM,

// 修改后
.stack_size = 8192,
.stack_policy = CLAW_TASK_STACK_INTERNAL_ONLY,
```

#### 文件
`esp-claw/components/claw_capabilities/cap_time/src/cap_time.c` 约第 404 行
